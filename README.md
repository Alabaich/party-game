# 🍻 Party Game

Інтерактивна party-гра: реєстрація через QR, ручний старт гри, рандомні
завдання з доказами (фото/відео), лідерборд із часом проходження і слайдшоу на ТВ.

## Стек
FastAPI · PostgreSQL · React (Vite) + Tailwind · Cloudflare R2 · Docker · Caddy

## Налаштування домену (DNS у GoDaddy)
Гра живе на субдомені, напр. `party.enjob.ca`. Створити A-запис:

1. Зайди на godaddy.com → **My Products** → біля домену `enjob.ca` натисни **DNS**
   (або **Manage DNS**).
2. **Add New Record** (Додати запис):
   - **Type:** `A`
   - **Name:** `party`  ← це субдомен (буде `party.enjob.ca`). Тільки слово, без домену.
   - **Value:** IP твого сервера (той самий, що для решти enjob.ca-сервісів).
     Дізнатись IP сервера: `curl -4 ifconfig.me`
   - **TTL:** 1 hour (або 600 секунд для швидшого поширення)
3. **Save**. Поширення DNS — від кількох хвилин до ~1 год.
4. Перевірка: `dig +short party.enjob.ca` має повернути IP сервера.

> Якщо такий субдомен уже керується Cloudflare/іншим NS — додавай запис там, де
> реально хостяться NS-сервери домену, а не в GoDaddy.

## Запуск
```bash
cp .env.example .env
nano .env                 # впиши GAME_DOMAIN, CORS_ORIGINS, секрети R2, пароль БД
docker compose up -d --build
```
Порти за замовчуванням: backend `8090`, frontend `8091` (обидва на 127.0.0.1,
перевірені вільними). Змінюються через `BACKEND_PORT` / `FRONTEND_PORT` у `.env`.

Далі — Caddy (він головний реверс-проксі на хості, тримає :80/:443 і робить SSL):
```bash
# додай блок із Caddyfile.snippet у /etc/caddy/Caddyfile, замінивши домен/порти
sudo nano /etc/caddy/Caddyfile
sudo caddy validate --config /etc/caddy/Caddyfile   # перевірка синтаксису
sudo systemctl reload caddy                          # застосувати без даунтайму
```
Caddy сам випустить Let's Encrypt сертифікат при першому запиті на домен.
QR-код має вести на `https://<GAME_DOMAIN>/`.

### Caddy через env (опційно)
Caddy на хості не читає `.env` проєкту. Якщо хочеш не хардкодити домен у Caddyfile,
використай плейсхолдер `{$GAME_DOMAIN}` у Caddyfile і експортуй змінну для сервісу:
```bash
sudo systemctl edit caddy
# додай:
# [Service]
# Environment=GAME_DOMAIN=party.enjob.ca
# Environment=BACKEND_PORT=8090
# Environment=FRONTEND_PORT=8091
sudo systemctl daemon-reload && sudo systemctl restart caddy
```
Тоді в Caddyfile: `{$GAME_DOMAIN} { handle_path /api/* { reverse_proxy 127.0.0.1:{$BACKEND_PORT} } ... }`.
Для разової вечірки простіше просто вписати домен явно.

## Як проходить гра
1. Гості скануют QR → реєстрація (імʼя + пʼють чи ні). UUID лягає в localStorage.
2. До старту кожен бачить екран «Чекаємо старту» зі списком гравців.
3. **Ти відкриваєш `/leaderboard` на своєму ПК і тиснеш «🚀 СТАРТ ГРИ».**
   У цей момент:
   - фіксується `started_at` (точка відліку часу);
   - кожному «вчасному» гравцю призначається **7 завдань** (≥1 кожного типу);
   - непитущі отримують максимум 1 алкогольне завдання.
4. Хто зареєструвався після старту — латекомер: 7 завдань лише типів 1/2, без пар.
5. Завдання закривається завантаженням медіа-доказу (можна замінити).
6. Коли гравець закрив усі 7 — фіксується його `completed_at`.
   **Час проходження = completed_at − started_at.**
7. Лідерборд: спершу всі, хто завершив усе (за часом, з 👑 топ-3), потім решта
   за кількістю виконаних.
8. Слайдшоу (`/slideshow`): фото 10 с, відео — на повну довжину.

## Як додавати/редагувати завдання
Завдання живуть у `backend/app/seed.py` (список `SEED_TASKS`). Формат рядка:
`(опис, тип, чи_алкогольне)`, де тип = `TaskType.SOLO` / `TARGETED` / `PAIRED`.
У TARGETED/PAIRED використовуй `{target}` — підставиться імʼя цілі/партнера.

Готуй список **до вечірки**. Сід заливається в БД один раз, коли таблиця tasks
порожня. Щоб перезалити після правок — очисти том БД:
```bash
docker compose down -v && docker compose up -d --build
```
Тримай достатньо НЕ-алкогольних завдань кожного типу, щоб ліміт для непитущих
завжди можна було задовольнити (зараз з запасом).

## Cloudflare R2 (важливо)
1. Створи bucket (напр. `party-media`).
2. Увімкни Public Access (r2.dev) або кастомний домен → це `R2_PUBLIC_BASE_URL`.
3. R2 API Token (Object Read & Write) → Access Key ID + Secret.
4. **CORS на бакеті** (фронт ллє напряму через presigned PUT):
   ```json
   [
     {
       "AllowedOrigins": ["https://party.enjob.ca"],
       "AllowedMethods": ["PUT", "GET"],
       "AllowedHeaders": ["Content-Type"],
       "MaxAgeSeconds": 3600
     }
   ]
   ```

## Тонкощі
- Старт гри ручний та ідемпотентний (повторне натискання нічого не псує),
  під блокуванням рядка — подвійного призначення не буде.
- Час на сервері — UTC; тривалість рахується на бекенді, від клієнтського годинника не залежить.
- Trust-based: захисту акаунтів нема (гра на один вечір). Кнопку СТАРТ бачать усі,
  але URL лідерборду ти нікому не даєш.

## Гарантії алгоритму призначення
Перевірено на 200+ симуляціях (13–14 гравців, різна частка непитущих):
- **Без дублів:** одне завдання не випадає гравцю двічі (ні соло, ні парне).
- **Покриття T1/T2:** кожне соло/таргет-завдання задіяне хоча б раз; розподіл
  рівномірний (нема такого, що одне завдання у всіх, а інше — orphaned).
- **Покриття T3:** усі парні завдання задіюються (кожне рівно раз). Гравець може
  потрапити у 2 пари — але з різними партнерами і різними завданнями.
  Алкогольні парні завдання ставляться лише в пари, де обидва п'ють.
- **Скільки парних завдань можна:** кожне T3-завдання = одна пара. Гравець може
  бути в кількох парах (з різними людьми, без повтору завдання). Ліміт участей на
  гравця = вільні слоти (TASKS_PER_USER − 2, бо 1 під соло і 1 під таргет). При 7
  завданнях це до 5 пар на гравця, тобто десятки парних завдань влазять без проблем.
  Перевірено: 9 і навіть 15 парних — повне покриття у 100% прогонів.
- **Алко-ліміт** для непитущих (≤1) має найвищий пріоритет і не порушується ніколи.
